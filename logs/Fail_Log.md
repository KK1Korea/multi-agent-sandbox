[16] 아키텍처 수정 — Observer가 토의 루프 전체 제어 + 오케스트레이터 경량화 — 2026-03-14 세션 #3 ★우선확인
────────────────────────────────────────

  ■ 증상:
    - [15]에서 오케스트레이터가 토의 루프를 운영하고 Observer 보고를 읽어 불균형을 판단
    - 문제: 오케스트레이터 = 메인 에이전트 = 사용자와 프로젝트를 진행하는 핵심
    - CPAS 설계 원칙: 메인 에이전트는 항상 깨끗한 컨텍스트로 프로젝트를 진행
    - 토의 원본 데이터가 오케스트레이터 컨텍스트에 들어가면 오염 + 토큰 낭비

  ■ 원인:
    - [15] 설계에서 오케스트레이터가 토의 루프를 직접 운영
      → 6턴 x 3섹션 = 최대 18턴의 원본 데이터가 오케스트레이터 컨텍스트에 축적
      → 메인 에이전트 컨텍스트 오염
    - Observer가 섹션 1~3을 종합하기 전까지 오케스트레이터는 토의 내용을 알 수 없는 게 맞음
    - Observer는 원래 CPAS 설계에서 토의 환경 제어 + 중립 분석의 겸용 역할
    - 불균형 감지도 Observer가 열세를 직접 감지하고 직접 조정하는 것이 자연스러움

  ■ 해결 — 역할 재분배:
    1. Observer의 역할 확장 (v0.5 → v0.6):
       - 기존: 토의 원본을 받아서 분석만
       - 변경: 토의 루프 전체를 직접 제어
         · Agent 도구 추가 (tools: ["Read", "Agent"])
         · Advocate/Skeptic 직접 소환 및 관리
         · 태그 제거를 직접 수행
         · 섹션 간 비교를 직접 수행
         · 열세 감지를 직접 수행
         · 최종 구조화된 보고서만 오케스트레이터에 반환

    2. 오케스트레이터 경량화 (SKILL.md v0.5 → v0.6):
       - Phase 1 (사전 준비): 워크스페이스 체크, 메모리 수집, 데이터 필터
       - Phase 2 (토의): Observer 1회 소환 + 모든 컨텍스트 전달 → 결과 대기
         · 오케스트레이터는 토의 데이터를 일절 보지 않음
         · Observer의 최종 보고서만 수신
       - Phase 3 (후처리): 품질 게이트, MasterLog 기록, 메모리 갱신

    3. Observer 프롬프트 구조 변경:
       - O-1-1: "환경을 제어한다: 에이전트 소환, 섹션 관리, 태그 제거" 추가
       - O-1-2: "오케스트레이터는 토의 데이터를 보지 않는다" 명시
       - O-1-4: 오케스트레이터로부터 받는 데이터 목록 (주제, 메모리, 필터 데이터, 평가)
       - O-2-4: "존재 노출 금지" → "내용 개입 금지" (환경 제어는 허용)
       - O-3-0-A: [교환 루프] 섹션 신설 — 에이전트 소환/재개, 태그 제거, 데이터 전달 절차
       - O-3-3-A: Observer가 직접 불균형 감지 (오케스트레이터 경유 아님)
       - X-0: "환경을 제어하되 내용은 절대 건드리지 않는다" 추가
       - X-X: "에이전트를 소환하고 관리하지만, 무엇을 말할지는 절대 지시하지 않는다"

  ■ 교훈:
    - 메인 에이전트(오케스트레이터)의 컨텍스트 청결 = CPAS 핵심 원칙
    - 토의 데이터는 Observer 컨텍스트 안에만 존재해야 함 — 메인 에이전트로 유출 금지
    - Observer의 "환경 제어"와 "내용 개입"은 완전히 다른 행위
      · 환경 제어: 누가 말하고, 어떤 능력으로 (허용)
      · 내용 개입: 무엇을 말하라 (금지)
    - [15]의 "관찰자가 관찰하고 오케스트레이터가 조정" → "관찰자가 관찰하고 직접 조정"으로 수정
    - 불균형 감지의 결정 주체와 실행 주체가 같아야 지연/오류 없음

[18] v0.6 → v0.7 — Observer 에이전트 호출 지시 구체화 (라이터 사용법) — 2026-03-14 세션 #3
────────────────────────────────────────

  ■ 증상:
    - [17]에서 Observer가 Agent 도구를 가지고 있었지만 한 번도 호출하지 않음
    - 원인 분석: Observer에게 "도구를 줬지만 사용법을 안 알려줬음"
    - 오케스트레이터 → Observer 전달 프롬프트: "Spawn advocate and skeptic agents as needed"만 있음
    - Observer 프롬프트 O-3-0-A: "Spawn (or resume) Advocate agent"로 추상적 지시만 있음
    - 실제 Agent 도구의 파라미터(subagent_type, prompt, resume 등) 문법이 어디에도 없음
    - LLM 입장에서 도구 호출 방법을 모르니 직접 텍스트 생성이 최소 저항 경로

  ■ 원인:
    - [16]에서 Observer에 Agent 도구를 부여할 때, 도구 자체만 줬지 "어떻게 쓰는지"를 안 알려줌
    - 비유: 라이터를 줬는데 켜는 법을 안 알려줘서, 장작 구해와서 나무 비벼서 불 붙인 격
    - Observer 프롬프트에 subagent_type 이름, prompt 구조, resume 패턴이 전무
    - 오케스트레이터의 Observer 호출 프롬프트에도 구체적 도구 사용법 없음

  ■ 해결 — v0.7 수정:
    1. Observer 프롬프트 O-3-0-A 전면 개정:
       - 최상단에 CRITICAL 경고:
         "반드시 Agent 도구로 소환. 직접 작성 절대 금지. 직접 쓰면 토의 전체 무효."
       - 평가자 소환 구체적 예시:
         · 첫 교환: subagent_type: "cpas-sandbox:advocate" + prompt 템플릿
         · 후속 교환: resume: {advocate_agent_id} + prompt 템플릿
         · 새 섹션: 새 에이전트 소환 (resume 금지)
       - 비평자 소환 구체적 예시:
         · 첫 교환: subagent_type: "cpas-sandbox:skeptic" + prompt 템플릿 (FILTERED_DATA 포함)
         · 후속 교환: resume: {skeptic_agent_id} + prompt 템플릿
       - 하단에 재확인:
         "모든 평가자/비평자 턴은 반드시 Agent 도구 호출. 토의 내용 직접 작성 금지."

    2. 오케스트레이터 SKILL.md Phase 2 프롬프트 개정:
       - "=== AGENT INVOCATION INSTRUCTIONS ===" 블록 추가
       - Agent 도구 파라미터 구체적 명시 (subagent_type, prompt, resume)
       - 평가자/비평자 각각의 소환/재개 문법 제시
       - "You produce ZERO debate content. You only manage the loop, strip tags, and analyze."
       - 상단에 CRITICAL 경고:
         "이 지시 없이는 Observer가 에이전트를 시뮬레이션함 → 3컨텍스트 아키텍처 붕괴"

    3. 한글 설계문서 동기화:
       - Prompts/Observer.md O-3-0-A 동일 구조로 한글 갱신
       - Agent 도구 호출 예시를 한글로 병기

  ■ 교훈:
    - "도구를 줬다 ≠ 사용법을 안다" — 도구 부여 시 반드시 호출 문법 + 예시를 같이 줘야 함
    - LLM은 도구 호출보다 직접 생성이 저항이 적음 → 구체적 호출 예시가 없으면 100% 직접 생성
    - [17]의 방안 A("프롬프트 강화")를 실행하되, 단순 "강제"가 아닌 "구체적 방법 제시"로 접근
    - [7]의 "아키텍처가 보장"에 보충: 도구 사용법도 아키텍처의 일부 — 도구가 있어도 사용법이 없으면 아키텍처 미완성
    - 다음 벤치마크에서 실제 에이전트 스폰 여부를 트랜스크립트 로그로 반드시 검증해야 함
    - ★ [20]에서 실패 입증: 실제 원인은 "사용법 부재"가 아니라 "서브에이전트 재귀 호출 불가(플랫폼 제한)"
