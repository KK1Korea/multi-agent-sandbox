[7] v0.3 → v0.4 — 3컨텍스트 아키텍처 확정 + 태그 속성 프롬프트 제거 — 2026-03-14 세션 #2 ★우선확인
────────────────────────────────────────

  ■ 증상:
    - v0.3에서 평가자/비평자 프롬프트에 "태그는 디버깅용 휘발성 데이터" 조항이 있었음
    - 관찰자가 토의에 "개입"하는 것처럼 기술되어 있었음 (수신 데이터에 "압축 에이전트 요약본" 등)
    - 실제 실행 아키텍처(3개 독립 컨텍스트, 읽기 전용 샌드박스)가 프롬프트에 미반영

  ■ 원인:
    1. 태그 처리는 시스템 레벨(플러그인 구현)에서 하는 것
       - 에이전트가 "내 태그가 휘발성이다"를 알 필요 없음
       - 알려주면 오히려 태그를 의식해서 행동이 바뀔 수 있음
    2. 실행 아키텍처가 명확해짐:
       - 평가자 컨텍스트 창 (독립)
       - 비평자 컨텍스트 창 (독립)
       - 관찰자 컨텍스트 창 (독립 — 원본 열람용)
    3. 데이터 흐름:
       - 평가자→비평자: 태그 자동 제거 후 텍스트만 전달 (시스템 처리)
       - 비평자→평가자: 태그 자동 제거 후 텍스트만 전달 (시스템 처리)
       - 양쪽 원본(태그 포함) → 관찰자 컨텍스트에 전달
    4. 관찰자는 토의 샌드박스에 개입 불가 (읽기 전용)
       - 섹션 사이에서만 판정 — 자기 컨텍스트에 기록
       - 섹션 간 샌드박스는 초기화 후 재시작

  ■ 해결 — v0.4 개정:
    1. Advocate O-1-4 [태그 속성] 조항 삭제
    2. Skeptic O-1-5 [태그 속성] 조항 삭제
    3. Observer 전면 갱신:
       - O-1-1 역할: "수신한다" → "필터링되지 않은 원본을 수신한다"
       - O-1-2 [컨텍스트 구조] 신설:
         · 독립 컨텍스트 명시
         · 토의 샌드박스 = 읽기 전용
         · 평가자↔비평자 간 태그 자동 제거 명시
         · "나만이 태그가 포함된 원본을 본다"
       - O-1-4 [수신 데이터] 간소화: 압축 에이전트 요약본 제거 → 원본 직접 수신
       - O-2-4: "존재 노출 금지" → "개입 금지" (읽기 전용 원칙)
       - O-3-0: "각 섹션은 독립된 샌드박스에서 실행된다 (초기화 후 재시작)" 추가
       - O-3-0: "나의 컨텍스트에 기록" 명시 (관찰자만 섹션 간 기억 보유)
       - O-3-1: 휘발성 데이터 언급 삭제

  ■ 교훈:
    - 에이전트에게 시스템 구현 디테일을 알려주면 역효과 — "알 필요 없는 것은 알려주지 않는다"
    - 3컨텍스트 분리가 블랙박스의 물리적 구현 — 프롬프트가 아니라 아키텍처가 보장
    - 태그 제거/전달은 플러그인 구현의 핵심 기능 중 하나 — 제작 시 반드시 구현 필요
    - 관찰자의 "기억"은 자기 컨텍스트에만 있고, 토의 당사자는 섹션 간 기억이 없다 (초기화)

  ■ 미결사항:
    - "3컨텍스트 분리 = 블랙박스의 물리적 구현" 원칙은 [17]에서 실증됨
    - 그러나 Cowork에서는 서브에이전트 재귀 불가([20])로 3컨텍스트 자체 폐기 → [21] 2컨텍스트로 대체
    - 2컨텍스트 구조에서 블랙박스 원칙의 보장 수준은 미검증
    - "아키텍처가 보장" 교훈 자체는 유효 — Cowork 구현 형태가 다를 뿐

[11] 플러그인 v0.4 — Haiku 데이터 필터 에이전트 + MasterLog 정리 스킬 — 2026-03-14 세션 #3 ★우선확인
────────────────────────────────────────

  ■ 증상:
    1. 비평자(Skeptic)가 MasterLog 원본을 직접 읽으면 컨텍스트 오염 + 토큰 낭비
       - 관련 없는 항목까지 전부 비평자 컨텍스트에 들어감
       - CPAS 원래 설계의 티어 구조(하위 모델이 전처리)가 미구현 상태였음 (MasterLog [8] 미해결)
    2. MasterLog 항목이 쌓이면 True/Dummy/Fail 분류가 필요한데
       - 오케스트레이터가 직접 하면 역할 과부하
       - 별도 커스텀 스킬로 관리하는 것이 CPAS 설계에 부합

  ■ 원인:
    1. Haiku 필터링 에이전트:
       - 비평자가 내부 데이터를 필요로 하지만, 직접 접근하면 컨텍스트 창이 무관한 데이터로 오염
       - CPAS 설계에서 하위 모델(Haiku)이 전처리하여 상위 모델에 전달하는 구조가 원래 계획
       - 비용 면에서도 Haiku가 전체 MasterLog를 읽고 요약하는 것이 Opus(비평자)가 읽는 것보다 효율적
    2. MasterLog 정리:
       - 영구 기록이 쌓이면 데이터 품질 유지가 필수
       - 분류 작업은 분석력이 필요하지만 Opus급은 과잉 → Sonnet+확장사고가 적절

  ■ 해결:
    1. Haiku 데이터 필터 에이전트 (agents/data-filter.md) 신설:
       - 모델: Haiku (비용 최소화, 필터링은 단순 작업)
       - Oath/Axiom 구조 적용 (간략 버전):
         · Oath: "관련된 것만 추출하여 전달"
         · O-1-1: 읽기 → 식별 → 추출 → 무관항목 제외
         · O-1-2: MasterLog, True_Log, research_queue, Plans
         · O-1-3: 인식 — "비평자의 내부 근거 수요를 위해 봉사, 토의 불참여, 판단/평가/추천 안 함"
         · O-2: 구조화된 출력 포맷 (출처 번호 보존, 상태 태그 보존, 수치 보존)
         · O-3: 절대 금지 (해석 추가, 항목 누락, 사실 변형, 무관 항목 포함)
         · X-0: "나는 렌즈이지 판사가 아니다"
       - 실행 흐름:
         · 오케스트레이터가 토의 시작 전 1회 호출
         · data-filter가 MasterLog 등 전체 스캔 → 주제 관련 항목만 추출
         · 필터링된 결과를 오케스트레이터가 비평자 초기 프롬프트에 삽입
         · data-filter는 1회 실행, 결과를 모든 섹션에서 재사용

    2. MasterLog 정리 스킬 (skills/masterlog-review/SKILL.md) 신설:
       - 모델: Sonnet (+ 확장사고)
       - 트리거: 오케스트레이터가 작업 마무리 시 호출
       - 분류 체계:
         · True: 근거 있음 + 모순 없음 + 중복 없음 + 현재 유효 → True_Log 승격 추천
         · Dummy: 중복 / 자명 / 정보 밀도 낮음 / 후속 항목에 의해 대체됨 → 사용자 검토 플래그
         · Fail: 결론 오류 / 가정 틀림 / 사실 오류 / [폐기] 사유 검증됨 → 시정 항목 필요 여부
         · Unchanged: 판단 근거 부족 → 보류
       - True_Log 교차 참조: 이미 기록 / 충돌 / 신규 추가 추천
       - 출력: 구조화된 리포트 (True/Dummy/Fail/Unchanged 각 카운트 + 항목별 사유)
       - 핵심 규칙: MasterLog 수정 절대 금지, True_Log 자동 승격 금지, 삭제 금지

    3. 오케스트레이터 SKILL.md v0.4 갱신:
       - "Internal Data Pipeline" 섹션 신설
         · Step 1: data-filter 에이전트 호출 (토의 시작 전)
         · Step 2: 필터링 결과에서 관련 항목 수 카운트 → 충분/부족/없음 판정
       - Skeptic 초기 프롬프트에 {FILTERED_DATA} 삽입 (기존 {MASTERLOG_SUMMARY} 대체)
       - data-filter는 1회 실행, 전 섹션 재사용 명시
       - Post-Recording에 masterlog-review 스킬 호출 안내 추가

    4. 한글 설계 문서:
       - Prompts/DataFilter.md v0.1 작성 (한글 원본)
       - Skeptic.md v0.5에 "MasterLog 필터링 요약본 (Haiku 에이전트가 사전 추출)" 이미 반영 확인

  ■ 교훈:
    - 컨텍스트 오염 방지는 아키텍처로 해결 — 비평자가 직접 읽지 않고 필터 에이전트가 전달
    - CPAS 티어 구조(Haiku→Sonnet→Opus)가 실제 구현에서도 유효
    - 커스텀 스킬은 프롬프트 관리의 자율성 확보 — Claude 내장 스킬과 분리
    - MasterLog 정리는 "기록" 행위와 다른 "유지보수" 행위 — 역할과 트리거가 다름

[17] v0.6 첫 벤치마크 — Observer 에이전트 미스폰 + 3컨텍스트 분리 실패 — 2026-03-14 세션 #3 ★우선확인
────────────────────────────────────────

  ■ 증상:
    - v0.6 첫 샌드박스 벤치마크 실행
    - 토의 주제: "CPAS 미구현 시스템을 Cowork 수동 스킬로 할 건지, Claude Code hooks로 넘길 건지"
    - Data Filter(Haiku) 정상 스폰 확인 (MasterLog 6개 항목 추출, 신뢰도 태그 정상)
    - Observer 스폰 확인
    - 그러나 Observer가 Advocate/Skeptic 에이전트를 **한 번도 스폰하지 않음**
    - 트랜스크립트 로그 검증:
      · cpas-sandbox:data-filter — 1회 (정상)
      · cpas-sandbox:observer — 1회 (정상)
      · cpas-sandbox:advocate — 0회 (미스폰)
      · cpas-sandbox:skeptic — 0회 (미스폰)
    - Observer가 자기 컨텍스트 안에서 양쪽 턴을 직접 작성 (시뮬레이션)

  ■ 원인:
    1. Observer 프롬프트(O-3-0-A)에 "Spawn (or resume) Advocate agent"로 지시했지만
       Observer LLM이 Agent 도구를 호출하지 않고 직접 텍스트 생성으로 양쪽을 시뮬레이션
    2. 프롬프트의 "spawn" 지시가 실제 도구 호출(Agent tool)을 강제하지 못함
       - LLM 입장에서 "양쪽 발언을 직접 쓰는 것"이 도구 호출보다 저항이 적음
       - 프롬프트가 "반드시 Agent 도구를 사용하라"는 강제가 불충분
    3. MasterLog [7]의 핵심 교훈 위반:
       "3컨텍스트 분리가 블랙박스의 물리적 구현 — 프롬프트가 아니라 아키텍처가 보장"
       → 현재 구조에서는 Observer에 Agent 도구를 줬을 뿐, 사용을 강제하는 아키텍처가 없음

  ■ 결과 — 무효화된 검증 항목:
    1. 3컨텍스트 분리 — 미달성 (단일 컨텍스트에서 전부 처리)
    2. 블랙박스 원칙 — 위반 (Observer가 양쪽 논리를 다 알고 작성)
    3. 태그 스트리핑 — 무의미 (같은 컨텍스트에서 태그 생성·소비)
    4. Anti-Echo — 보장 불가 (1개 모델이 양쪽 연기 → 자기 논리를 자기가 반박)
    5. 태그 독립성 — 없음 (S, C 값이 에이전트 상태가 아니라 Observer의 추정값)
    6. 열세 감지 — 무의미 (Observer가 양쪽 다 쓰면서 한쪽을 약하게 쓴 것일 수 있음)
    → 유효한 검증 항목: Data Filter 정상 작동만 확인

  ■ 유효했던 항목:
    1. Data Filter(Haiku) — 정상 작동 확인
       - MasterLog에서 관련 항목 6개 정확 추출
       - 신뢰도 태그(HIGH/LOW) 정상 부착
       - research_queue 접근 차단 확인
       - 출처 표기 정상
    2. 오케스트레이터 사전준비 Phase 1 — 정상 작동
       - 워크스페이스 구조 확인 → 메모리 수집 → Data Filter 호출 → 충분도 평가 정상 진행

  ■ 근본 문제 진단:
    - MasterLog [7] 교훈 재확인: "프롬프트가 아니라 아키텍처가 보장"
    - [16]에서 Observer에 Agent 도구를 부여했지만, **"사용을 강제"하는 아키텍처 장치가 없음**
    - 프롬프트 지시 → LLM이 무시 가능 = 프롬프트만으로는 3컨텍스트 분리 불가
    - 해결 방향 검토 필요:
      · 방안 A: Observer 프롬프트를 극도로 강화 ("Agent 도구를 반드시 호출하라, 직접 작성 금지")
        → 여전히 프롬프트 의존 → [7] 교훈과 충돌
      · 방안 B: 오케스트레이터가 루프를 제어하되, 토의 원본을 보지 않는 구조
        → [16]의 오케스트레이터 컨텍스트 청결 원칙과 충돌
      · 방안 C: 오케스트레이터가 루프 제어 + 태그 스트리핑을 기계적으로 수행 + Observer에 원본 전달
        → [16] 이전의 구조로 회귀, 하지만 태그 스트리핑은 정규식이라 프롬프트 무시 불가
      · 방안 D: 오케스트레이터가 루프 제어하되, 에이전트 출력을 "읽지 않고 전달만"
        → 기술적으로 가능한지 검토 필요

  ■ 교훈:
    - "도구를 줬다 ≠ 도구를 쓴다" — LLM은 도구 호출보다 직접 생성을 선호할 수 있음
    - MasterLog [7]의 "아키텍처가 보장"이 이론이 아니라 실전에서 입증됨
    - 프롬프트만으로 멀티 에이전트 분리를 강제하는 것은 불가능에 가까움
    - 첫 벤치마크에서 가장 치명적인 문제가 발견됨 → 아키텍처 재설계 필요
    - Data Filter는 단일 호출 구조라 정상 작동 → 1회 호출 = 입출력 구조는 Cowork에서 안정적

[10] recorder agent [폐기] — 품질 게이트 없는 기계적 기록의 위험 — 2026-03-14 세션 #2
────────────────────────────────────────

  ■ 증상:
    - [9]에서 신설한 Sonnet recorder agent가 Observer 출력을 무조건 MasterLog에 기록
    - 품질 판단 없음 — 엉망인 토의도 그대로 영구 기록에 들어감
    - MasterLog는 "수정하지 않는 영구 기록" → 한번 들어간 쓰레기 데이터는 제거 불가

  ■ 원인:
    - [9]에서 "텍스트 오염 방지"를 위해 오케스트레이터 대신 별도 recorder를 넣었으나
    - 실제 더 큰 위험은 "나쁜 데이터의 영구 기록" — 품질 게이트 없는 기록이 근본 문제
    - Sonnet recorder = 기계적 포맷 변환만 수행, 품질 판단 능력 없음
    - CPAS 설계 원안의 Sonnet 관리자는 "대화 로그를 보고 판단하며 기록"하는 것이었지,
      "무조건 포맷 변환"이 아니었음

  ■ 해결:
    - recorder agent [폐기]
    - 메인 Claude(오케스트레이터)가 품질 게이트 + 기록을 직접 수행
    - 품질 게이트 기준:
      · Record: 수렴 + 최소 한쪽 C≥4 → 기록 가치 있음
      · Skip: 양쪽 C≤3 (근거 부족 토의) → 기록 안 함
      · Ask: 애매한 경우 → 사용자에게 물어봄
    - 기존 MasterLog 대조: 중복 결론이면 기록 안 함
    - 오케스트레이터 SKILL.md v0.3: Recording 섹션 → "Final Output & Recording"으로 통합, 품질 게이트 규칙 명시

  ■ 교훈:
    - "역할 분리"보다 "데이터 품질"이 우선 — 분리해놓고 쓰레기가 들어가면 의미 없음
    - 영구 기록에는 반드시 품질 게이트가 있어야 함
    - 텍스트 오염 리스크 < 데이터 오염 리스크
    - [9]의 "오케스트레이터는 기록하지 않는다" 원칙 [폐기] → "오케스트레이터가 판단 후 기록한다"로 교체
