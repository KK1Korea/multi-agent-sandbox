# CPAS Project Review & Data Quality Management

프로젝트 로그 전체를 정리하고, current_task.md를 최신 상태로 유지하는 스킬.
로그 분류 + 구형 데이터 Dummy 이동 + 버전 추적 + 진행도 갱신을 한 번에 수행한다.

## When to Use

- 세션 마무리 시 (새 항목이 추가되었거나 진행 상태가 변경된 경우)
- 사용자가 `/review` 명령 실행 시
- 프로젝트 버전이 올라갈 때 (기능 추가, 아키텍처 변경 등)
- sandbox-orchestrator Step 0.5에서 stale 데이터 감지 후 사용자가 승인한 경우
- 대량의 MasterLog 항목이 쌓였을 때

## Prerequisites

Read the following files:
- `current_task.md` — 프로젝트 라이브 상태 + 버전 히스토리
- `MasterLog.md` — 미분류 항목 (스테이징 영역)
- `True_Log.md` — 실증 검증된 사실
- `Dummy_Log/Dummy_Log_*.md` — 최신 Dummy 로그 파일
- `Fail_Log.md` — 실패 입증된 항목
- `CLAUDE.md` — 프로젝트 규칙

## 파일 구조

```
CPAS/
├── current_task.md       ← 라이브 상태 + 버전 히스토리 (이 스킬이 관리)
├── MasterLog.md          ← 미분류 항목만 (스테이징 영역)
├── True_Log.md           ← 실증 성공 항목 (자동 승격)
├── Fail_Log.md           ← 실패 입증 항목 (자동 이동)
└── Dummy_Log/
    ├── Dummy_Log_1.md    ← 저가치/중복/구형 항목 (1000줄 로테이션)
    ├── Dummy_Log_2.md    ← 1000줄 초과 시 새 파일
    └── ...
```

## Phase 1 — current_task.md 버전 갱신

### Step 1.1 — 버전 확인

current_task.md의 `[작업 버전]` 섹션을 읽는다.
- 마지막 버전 번호 확인
- 이번 세션에서 버전 변경이 있었는가? (기능 추가, 아키텍처 변경, 플러그인 업데이트 등)
- 버전 변경이 있으면 → 새 버전 엔트리 추가
- 버전 변경이 없으면 → 기존 버전의 요약만 갱신

### Step 1.2 — 버전 엔트리 추가

새 버전이 있을 경우 `[작업 버전]` 섹션에 추가:

```
## [작업 버전]
v0.9.3 : masterlog-review 모델 고정 제거, ET 사용자 승인, stale check, 프로젝트 관리 분리
v0.9.2 : Data-Filter 3분할, 벤치마크 2회 완료, 볼륨 분할 규칙
v0.9.1 : Data-Filter 3분할 초안
v0.9.0 : Observer 제거, 2-level 아키텍처, Cowork_CPAS 분리
...
```

**규칙:**
- 최신이 위 (역순)
- 한 줄 요약 (핵심 변경만)
- 절대 이전 버전 내용을 수정하지 않음 (쌓기만)
- 버전 번호는 프로젝트/플러그인 버전을 따름

### Step 1.3 — 라이브 상태 갱신

current_task.md의 나머지 섹션들을 현재 상태로 갱신:

1. `[현재 상황]` — 현재 Phase, 주요 상태 한 줄
2. `[지금 해야 할 일]` — 현재 우선순위 작업 목록
   - ★ 이 섹션이 샌드박스 토의의 `{MEMORY_CONTEXT}`로 전달됨
   - 토의가 쟁점에서 벗어나지 않도록 현재 작업 목표를 명확하게 기술
3. `[로그 현황]` — 각 로그의 항목 번호 목록
4. `[진행도]` — Phase별 퍼센트 바
5. `[최종 갱신]` — 날짜 + 세션 번호

## Phase 2 — MasterLog 분류

### Step 2.1 — 전체 항목 스캔

MasterLog.md의 모든 항목을 읽는다. 각 항목에 대해 평가:

1. **상태 태그 확인**: 항목의 태그가 분류 방향을 시사한다.
   - `[확정]` → True 후보 (실증 검증 필요)
   - `[폐기]` → Fail 또는 Dummy 후보 (폐기 사유에 따라 분류)
   - `[구상]` → 미분류 유지 (검증 전)
   - `[진행]` → 미분류 유지 (작업 중)
   - `[보류]` → 미분류 유지 (부활 조건 미충족)
2. **실증 여부**: 벤치마크, 테스트, 로그 검증 등 경험적 검증이 있는가?
3. **결론의 성패**: 항목이 기술한 접근법/결론이 실제로 성공했는가, 실패했는가?
4. **현재 유효성**: 후속 항목에 의해 대체/무효화되었는가?
5. **정보 밀도**: 실질적 교훈이 있는가, 단순 기록에 불과한가?
6. **버전 연관**: `[작업 버전]`과 대조 — 이 항목이 속한 버전이 현재 유효한가?

**★ 태그와 분류의 관계:**
  - `[확정]`이라도 실증 증거가 없으면 True 불가 (태그 ≠ 자동 분류)
  - `[구상]`이라도 벤치마크 테스트 결과 등 부가 데이터가 있으면 True 후보 검토
  - `[폐기]`의 폐기 사유가 후속 항목에서 입증됐으면 Fail, 단순 대체면 Dummy

### Step 2.2 — 분류

**★★★ 핵심 판정 기준: "기록이 정확한가?"가 아니라 "결론이 실증으로 검증됐는가?" ★★★**

#### True — 실증 성공 + 지속 참조 가치 (자동 → True_Log.md)

항목이 다음을 **모두** 충족:
  - 결론이 **경험적 증거**(벤치마크, 테스트, 로그 검증)로 **성공이 입증**됨
  - 후속 항목에 의해 모순되지 않음
  - 다른 항목과 중복이 아님
  - 현재 프로젝트 상태에서 여전히 유효
  - **★ 지속 참조 가치**: 향후 의사결정에서 계속적으로 참조할 수 있는 핵심 검증사항인가?

→ **자동 처리**: True_Log.md에 추가 + MasterLog.md에서 제거

**True 판정 금지 사례:**
  - 단순히 문서화가 정확한 것 (기록 정확성 ≠ 결론 검증)
  - 설계 결정이지만 아직 테스트 안 된 것
  - 실패를 기록한 항목 (실패 진단이 정확해도 → 진단의 실증이 별도로 필요)
  - 구버전 설계가 후속 버전에 완전 대체된 것 (→ Dummy)

**★ True_Log 잔류 기준 (기존 항목 재평가 시에도 적용):**
  - True_Log에는 **핵심 검증사항 + 계속적으로 참조 가능한 것만** 잔류
  - 구버전/대체된 항목 → Dummy (대체 항목 번호 명시)
  - 실패가 확인된 항목 → Fail (실패 입증 항목 번호 명시)
  - "검증됐지만 후속 버전이 상위 호환" → Dummy (핵심 교훈이 후속에 흡수됨)

#### Fail — 실패 입증 (자동 → Fail_Log.md)

항목이 다음 중 **하나라도** 해당:
  - 항목이 기술한 접근법이 후속 테스트/벤치마크에서 **실패**함
  - 결론이 후속 증거에 의해 **틀린 것으로 입증**됨
  - 기반 가정이 **거짓으로 판명**됨
  - 사실 오류가 포함됨

→ **자동 처리**: Fail_Log.md에 추가 + MasterLog.md에서 제거
→ Fail_Log 항목에는 **실패 사유 + 어떤 후속 항목이 실패를 입증했는지** 반드시 기재

**★ 중요: 실패 진단 항목 vs 실패한 접근법 항목 ★**
  - "X가 실패했다"는 진단 기록 → 진단 자체의 실증이 있으면 True 후보
  - "X를 하자"고 확정한 원본 항목 → 후속에서 X가 실패하면 Fail
  - 핵심: **실패한 접근법을 제안/확정한 항목이 Fail. 실패를 정확히 진단한 항목은 별개.**

#### Dummy — 저가치/중복/대체됨 (자동 → Dummy_Log/)

항목이 다음 중 **하나라도** 해당:
  - 다른 항목과 중복 (어떤 항목인지 명시)
  - 결론이 자명/사소함
  - 정보 밀도가 낮음 (많은 텍스트, 적은 실질 내용)
  - 후속 항목에 의해 완전히 대체됨 (대체 항목 명시)
  - **구버전 설계**: 후속 버전이 상위 호환이며, 핵심 교훈이 후속에 흡수됨
  - **True_Log/Fail_Log 재평가 대상**: 기존에 분류됐으나 위 조건에 해당하게 된 항목

→ **자동 처리**: 최신 Dummy_Log 파일에 추가 + 원본 로그에서 제거
→ Dummy_Log 파일이 1000줄 초과 시 새 파일 생성 (Dummy_Log_2.md, Dummy_Log_3.md...)

#### 미분류 — MasterLog에 잔류

  - 아직 실증 증거가 없는 설계 결정
  - 판단 근거가 불충분한 항목
  - 최근 추가되어 검증 시간이 부족한 항목

→ **처리 없음**: MasterLog.md에 그대로 유지
→ **경과 규칙**: 3세션 이상 미분류 상태로 남은 항목 → Dummy 후보로 플래그

## Phase 3 — True_Log / Fail_Log 재평가

**★ 이 Phase가 핵심 — MasterLog만 분류하는 게 아니라, 기존 True/Fail도 재평가한다 ★**

### Step 3.1 — 버전 기반 구형 데이터 탐지

`[작업 버전]`의 히스토리와 True_Log/Fail_Log 항목을 대조:

1. 각 항목이 어느 버전 시점에 작성됐는지 확인 (세션 번호 → 버전 매핑)
2. 현재 버전에서 해당 항목의 결론이 여전히 유효한지 판단
3. **구형 판정 기준:**
   - 항목이 기술한 버전이 2+ 메이저 버전 이전이고
   - 후속 항목이 동일 영역을 다루며
   - 핵심 교훈이 후속에 흡수된 경우
   → Dummy 후보

### Step 3.2 — True_Log 재평가

True_Log의 모든 항목에 대해:
- **잔류**: 핵심 검증사항 + 현재 버전에서 계속 참조 가능
- **Dummy 이동**: 구버전/대체됨 — `[Dummy 사유]` 헤더 부착 후 Dummy_Log로 이동
- **Fail 이동**: 후속 검증에서 실패 확인됨 — `[Fail 사유]` 헤더 부착 후 Fail_Log로 이동

### Step 3.3 — Fail_Log 재평가

Fail_Log의 모든 항목에 대해:
- **잔류**: 실패 교훈이 현재 버전에서 여전히 유효 (같은 실수 방지)
- **Dummy 이동**: 실패한 접근법이 완전히 폐기되어 재발 가능성 없음

## Phase 4 — 자동 이동 실행

**이 단계가 핵심 — 분류 후 실제 파일 조작을 수행한다.**

1. True 항목: True_Log.md에 **원본 그대로** 추가 (누락/압축/요약 절대 금지)
2. Fail 항목: Fail_Log.md에 **원본 그대로** 추가 + 실패 사유 헤더 부착
3. Dummy 항목: 최신 Dummy_Log 파일에 **원본 그대로** 추가 + Dummy 사유 헤더 부착
4. 원본 로그에서 이동된 항목 제거
5. MasterLog.md 헤더의 "이전 로그" 인덱스 갱신
6. current_task.md의 `[로그 현황]` 갱신

## 볼륨 분할 규칙 (1500줄 제한)

모든 로그 파일은 **1파일 = 최대 1500줄** 규칙을 따른다.
항목 이동 전에 대상 파일의 줄 수를 확인하고, 초과 시 새 볼륨에 기록한다.

- **True_Log**: 1500줄 초과 → True_Log_1.md, True_Log_2.md...
- **Fail_Log**: 1500줄 초과 → Fail_Log_1.md, Fail_Log_2.md...
- **Dummy_Log**: 기존 규칙 유지 (1000줄 로테이션)
- **MasterLog**: 볼륨 분할 안 함. 초과분 → Dummy로 강제 이동.

### 분할 시 확인 사항

1. 이동 전: `wc -l` 등으로 대상 파일의 현재 줄 수 확인
2. 1500줄 초과 예상 시: 새 볼륨 파일 생성 후 거기에 기록
3. 새 볼륨 파일 헤더: 원본 파일 헤더 복사 + "Volume N" 표기
4. 분할 사실을 결과 보고(Phase 5)에 포함

## Important Rules

- **원본 보존 절대 원칙**: 항목 이동 시 내용을 누락/압축/요약하지 않는다. 글자 하나 바꾸지 않는다.
- **자동 이동**: 분류 즉시 해당 로그로 이동한다. 사용자 승인 불필요.
- **Dummy_Log 로테이션**: 현재 Dummy_Log 파일이 1000줄 초과 시 새 번호 파일 생성.
- **MasterLog = 스테이징**: 미분류 항목만 남긴다. 분류된 항목은 반드시 이동.
- **경과 규칙**: 3세션 이상 미분류 → Dummy 후보 플래그 (다음 리뷰에서 재평가).
- **MasterLog 용량 제한**: 최대 ~1500줄. 초과 시 가장 오래된 미분류 항목부터 Dummy로 강제 이동.
- **Fail ≠ True**: 실패한 접근법은 절대 True로 보내지 않는다.
- **버전 추적**: current_task.md의 `[작업 버전]`은 쌓기만 하고 수정하지 않는다.

## 범위 제한

- 이 스킬의 작업 범위는 **로그 파일 분류/이동 + current_task.md 갱신**만 해당한다.
- GitHub push, 플러그인 재패키징, 외부 동기화 등은 이 스킬의 범위 밖이다.
- 샌드박스 토의 실행은 이 스킬의 범위 밖이다 (cpas-sandbox 플러그인 담당).
- 범위 밖 작업이 필요하면 결과 보고에 "추가 작업 필요" 항목으로 명시하고 **실행하지 않는다.**
