# 최종 갱신: 2026-03-14 세션 #4

================================================================================
0. 배경 — 왜 Cowork_CPAS인가
================================================================================

[CPAS 본체 아키텍처]
  메인(오케스트레이터) → Observer(서브) → {Advocate, Skeptic}(서브의 서브)
  · 3단계 중첩. Observer가 토의 루프 제어 + 분석.
  · 오케스트레이터 컨텍스트 완전 청결 — 토의 데이터 미노출.
  · Claude Code 전용 — 서브에이전트 재귀 호출 가능 시 사용.

[Cowork 플랫폼 제한]
  "Subagents cannot spawn other subagents" — 공식 확정 (MasterLog [20])
  · 최대 2단계: 메인 → 서브에이전트
  · Observer(서브)가 Advocate/Skeptic(서브의 서브) 소환 불가
  · tools: ["Read", "Agent"] 선언해도 서브에이전트에서 Agent 도구 런타임 미바인딩

[Cowork_CPAS 방향]
  · Observer 제거. 메인 에이전트가 토의 루프 제어 + 관찰 + 판단 직접 수행.
  · 2단계만 사용: 메인 → {Advocate, Skeptic, Data-Filter}
  · 컨텍스트 오염이 아님: 토의 내용을 보되 건드리지 않음. 관찰과 판단의 겸용.
  · "토의를 건드리는 게 아니니까" — 사용자 판단 근거

1. 아키텍처 비교
================================================================================

[CPAS 본체 — 3단계 (이상적)]
  ┌──────────────┐
  │ Orchestrator  │  ← 프로젝트 진행, 컨텍스트 청결
  │ (Main Agent)  │
  └──────┬───────┘
         │ 1회 소환
  ┌──────▼───────┐
  │   Observer    │  ← 토의 루프 제어, 태그 분석, 환경 조정
  │ (Subagent 1)  │
  └──┬───────┬───┘
     │       │ 소환/재개
  ┌──▼──┐ ┌─▼────┐
  │Adv. │ │Skep. │  ← 독립 컨텍스트 토의
  │(Sub2)│ │(Sub2)│
  └─────┘ └──────┘

  장점: 오케스트레이터 컨텍스트 완전 청결
  단점: 3단계 중첩 필요 → Cowork 불가

[Cowork_CPAS — 2단계 (최적화)]
  ┌──────────────────────────┐
  │     Orchestrator          │  ← 프로젝트 진행 + 토의 루프 제어
  │     (Main Agent)          │     + 태그 분석 + 판단
  │                           │
  │  ┌─ Observer 기능 흡수 ─┐ │
  │  │ · 태그 스트리핑       │ │
  │  │ · 섹션 관리           │ │
  │  │ · 수렴 판정           │ │
  │  │ · 불균형 감지/조정    │ │
  │  │ · 쟁점 구조화         │ │
  │  │ · 최종 보고서 작성    │ │
  │  └───────────────────────┘ │
  └──┬──────┬──────┬──────────┘
     │      │      │ 직접 소환
  ┌──▼──┐┌─▼────┐┌▼─────────┐
  │Adv. ││Skep. ││DataFilter│
  │(Sub)││(Sub) ││(Sub/Haiku│
  └─────┘└──────┘└──────────┘

  장점: Cowork 플랫폼 호환, 단순한 구조
  단점: 오케스트레이터 컨텍스트에 토의 데이터 축적
        (단, 토의 내용에 개입하지 않으므로 오염 아님)

[흡수되는 기능 — Observer.md 기준]

  Observer 기능              | Cowork_CPAS에서          | 비고
  ─────────────────────────|─────────────────────────|──────────────
  O-1-1 에이전트 소환/관리   | 오케스트레이터 직접 수행  | Agent 도구 사용
  O-1-2 독립 컨텍스트        | 제거 (공유 컨텍스트)     | 트레이드오프 감수
  O-3-0 섹션 구조 관리       | 오케스트레이터 직접 수행  | 3교환×3섹션 유지
  O-3-0-A 교환 루프 실행     | 오케스트레이터 직접 수행  | 소환/재개/태그제거
  O-3-1 태그 읽기 기준       | 오케스트레이터 직접 수행  | D/R/C/A/S 분석
  O-3-2 수렴 판정            | 오케스트레이터 직접 수행  | S≤4 기준 유지
  O-3-3 문제 감지            | 오케스트레이터 직접 수행  | 토의 이탈/과열 감지
  O-3-3-A 불균형 감지/조정   | 오케스트레이터 직접 수행  | 불균형 보고
  O-3-4 섹션 간 비교         | 오케스트레이터 직접 수행  | 조기종료 판정
  O-4 최종 보고서            | 오케스트레이터 직접 작성  | 구조화된 쟁점 보고

[변경되는 원칙 — Observer O-2 절대금지]

  원래 Observer 규칙         | Cowork_CPAS에서          | 사유
  ─────────────────────────|─────────────────────────|──────────────
  O-2-1 승패 판정 금지       | 완화 — 근거 기반 평가 허용| 사용자 대리인
  O-2-2 의견 제시 금지       | 완화 — 분석 기반 평가 허용| 사용자 대리인
  O-2-3 결론 도출 금지       | 완화 — 권고 가능          | 최종 결정은 사용자
  O-2-4 내용 개입 금지       | 유지 — 절대 금지          | 핵심 원칙 불변

  핵심: "환경 제어 + 내용 불간섭"은 유지. 다만 분석 후 평가/권고는 허용.
  사유: Observer는 중립 계층이었지만, 오케스트레이터는 사용자의 대리인.
  → "누가 더 강한 근거를 가졌는가"를 말할 수 있음. "무엇을 말하라"는 불가.

4. Cowork_CPAS 오케스트레이터 설계
================================================================================

[Phase 1 — 사전 준비] (기존과 동일)

  Step 0. 워크스페이스 구조 확인
    · current_task.md, MasterLog.md, True_Log.md, Fail_Log.md, Dummy_Log/, CLAUDE.md
    · 미존재 시 workspace-init 스킬 실행

  Step 1. 메모리 + 라이브 상태 수집
    · Claude 메모리 (내장)
    · current_task.md 읽기
    · → {MEMORY_CONTEXT} 조합

  Step 2. Data-Filter 소환 (Haiku)
    · data-filter 에이전트 소환
    · True_Log (HIGH), Fail_Log (HIGH), MasterLog (MEDIUM) 읽기
    · → {FILTERED_DATA} 수신

  Step 3. 내부 데이터 충분성 평가
    · 2+ 관련 항목 → sufficient
    · 0-1 관련 항목 → thin
    · "No relevant data" → search-only

[Phase 2 — 토의 루프] (핵심 변경)

  오케스트레이터가 직접 실행. Observer 없음.

  섹션 구조: 1섹션 = 3교환(6턴). 최대 3섹션.

  교환 루프 (섹션 N):

    교환 1:
      1. Advocate 소환 (subagent_type: "cpas-sandbox:advocate")
         · MEMORY_CONTEXT + 토의 주제 + "찬성 입장" 전달
      2. Advocate 원본 출력 저장 (태그 포함)
      3. 태그 라인 추출: [D-?] [R-?] [C-?] [A-?] [S-?]
      4. 태그/구분선 제거 → 순수 텍스트
      5. Skeptic 소환 (subagent_type: "cpas-sandbox:skeptic")
         · MEMORY_CONTEXT + 토의 주제 + Advocate 순수 텍스트
         + FILTERED_DATA + 충분성 평가 전달
      6. Skeptic 원본 출력 저장 (태그 포함)
      7. 태그 라인 추출
      8. 태그/구분선 제거 → 순수 텍스트 (다음 교환 Advocate용)

    교환 2-3:
      · Advocate resume (같은 agent_id) + Skeptic 순수 텍스트
      · Skeptic resume (같은 agent_id) + Advocate 순수 텍스트
      · 매 턴 태그 추출 + 원본 저장

    섹션 종료 후:
      · 태그 시계열 분석 (6턴 전체)
      · 수렴 판정 (S≤4 기준)
      · 불균형 감지 (S/C 격차)

  섹션 간 전이:

    섹션 1 완료:
      · 쟁점 구조화
      · 불균형 감지 → 섹션 2에서 열세 측 불균형 보고
      · 새 Advocate/Skeptic 소환 (resume 금지 — 신선한 컨텍스트)

    섹션 2 완료:
      · 쟁점 구조화
      · 섹션 1 결론과 비교
      · 동일 결론 → 섹션 3 생략. 최종 보고서 작성.
      · 상이 결론 → 섹션 3 진행.

    섹션 3 완료 (해당 시):
      · 최종 쟁점 구조화
      · 3섹션 걸친 변화 추이 기록

[Phase 3 — 후처리] (확장)

  Step 1. 구조화된 분석 보고서 작성
    · 기존 Observer O-4 포맷 사용 (쟁점/근거/차이/미검증/분리 쟁점)
    · + 오케스트레이터 평가 추가:
      [오케스트레이터 평가]
      · 근거 강도: 어느 측이 더 강한 근거를 제시했는가
      · 실행 가능성: 현재 프로젝트 상태에서 실현 가능한가
      · 권고: 사용자에게 제안하는 방향 (최종 결정은 사용자)

  Step 2. 품질 게이트 (기존과 동일)
    · Record / Skip / Ask user

  Step 3. MasterLog 기록 (기존과 동일)
    · 태그: [구상]

  Step 4. 메모리 갱신 (기존과 동일)
    · 사용자 승인 후 갱신

5. 컨텍스트 비용 분석
================================================================================

[CPAS 본체 — 오케스트레이터 컨텍스트 사용량]
  · Phase 1 (준비): ~2K 토큰
  · Phase 2 (토의): 0 (Observer에 위임)
  · Phase 3 (후처리): ~1K 토큰 (Observer 보고서만 수신)
  · 합계: ~3K 토큰

[Cowork_CPAS — 오케스트레이터 컨텍스트 사용량]
  · Phase 1 (준비): ~2K 토큰
  · Phase 2 (토의): ~12-18K 토큰 (6턴/섹션 × 최대 3섹션)
    · 각 턴 ~500-1000 토큰 (원본 + 태그)
    · 3섹션 최대: 18턴 × ~1K = ~18K
  · Phase 3 (후처리): ~2K 토큰 (분석 + 보고서)
  · 합계: ~16-22K 토큰

  추가 비용: ~13-19K 토큰
  허용 범위: Cowork 컨텍스트 윈도우 대비 소량. 세션 진행에 지장 없음.

  완화 전략:
  · 섹션 2에서 조기종료 시 ~12K로 감소
  · 이전 섹션 원본 데이터를 요약으로 압축 가능 (구조화 완료 후)
  · 단, 현재 진행 중인 섹션의 데이터는 원본 유지 필수
