0. 배경 — 왜 Cowork_CPAS인가
================================================================================

[CPAS 본체 아키텍처]
  메인(오케스트레이터) → Observer(서브) → {Advocate, Skeptic}(서브의 서브)
  · 3단계 중첩. Observer가 토의 루프 제어 + 분석.
  · 오케스트레이터 컨텍스트 완전 청결 — 토의 데이터 미노출.
  · Claude Code 전용 — 서브에이전트 재귀 호출 가능 시 사용.

[Cowork 플랫폼 제한]
  "Subagents cannot spawn other subagents" — 공식 확정 (MasterLog [20])
  · 최대 2단계: 메인 → 서브에이전트
  · Observer(서브)가 Advocate/Skeptic(서브의 서브) 소환 불가
  · tools: ["Read", "Agent"] 선언해도 서브에이전트에서 Agent 도구 런타임 미바인딩

[Cowork_CPAS 방향]
  · Observer 제거. 메인 에이전트가 토의 루프 제어 + 관찰 + 판단 직접 수행.
  · 2단계만 사용: 메인 → {Advocate, Skeptic, Data-Filter}
  · 컨텍스트 오염이 아님: 토의 내용을 보되 건드리지 않음. 관찰과 판단의 겸용.
  · "토의를 건드리는 게 아니니까" — 사용자 판단 근거

[CPAS 본체 — 3단계 (이상적)]
  ┌──────────────┐
  │ Orchestrator  │  ← 프로젝트 진행, 컨텍스트 청결
  │ (Main Agent)  │
  └──────┬───────┘
         │ 1회 소환
  ┌──────▼───────┐
  │   Observer    │  ← 토의 루프 제어, 태그 분석, 환경 조정
  │ (Subagent 1)  │
  └──┬───────┬───┘
     │       │ 소환/재개
  ┌──▼──┐ ┌─▼────┐
  │Adv. │ │Skep. │  ← 독립 컨텍스트 토의
  │(Sub2)│ │(Sub2)│
  └─────┘ └──────┘

  장점: 오케스트레이터 컨텍스트 완전 청결
  단점: 3단계 중첩 필요 → Cowork 불가

[Cowork_CPAS — 2단계 (최적화)]
  ┌──────────────────────────┐
  │     Orchestrator          │  ← 프로젝트 진행 + 토의 루프 제어
  │     (Main Agent)          │     + 태그 분석 + 판단
  │                           │
  │  ┌─ Observer 기능 흡수 ─┐ │
  │  │ · 태그 스트리핑       │ │
  │  │ · 섹션 관리           │ │
  │  │ · 수렴 판정           │ │
  │  │ · 불균형 감지/보고    │ │
  │  │ · 쟁점 구조화         │ │
  │  │ · 최종 보고서 작성    │ │
  │  └───────────────────────┘ │
  └──┬──────┬──────┬──────────┘
     │      │      │ 직접 소환
  ┌──▼──┐┌─▼────┐┌▼─────────┐
  │Adv. ││Skep. ││DataFilter│
  │(Sub)││(Sub) ││(Sub/Haiku│
  └─────┘└──────┘└──────────┘

  장점: Cowork 플랫폼 호환, 단순한 구조
  단점: 오케스트레이터 컨텍스트에 토의 데이터 축적
        (단, 토의 내용에 개입하지 않으므로 오염 아님)

4. Cowork_CPAS 오케스트레이터 설계
================================================================================

[Phase 1 — 사전 준비] (기존과 동일)

  Step 0. 워크스페이스 구조 확인
    · current_task.md, MasterLog.md, True_Log.md, Fail_Log.md, Dummy_Log/, CLAUDE.md
    · 미존재 시 workspace-init 스킬 실행

  Step 1. 메모리 + 라이브 상태 수집
    · Claude 메모리 (내장)
    · current_task.md 읽기
    · → {MEMORY_CONTEXT} 조합

  Step 2. Data-Filter 소환 (Haiku)
    · data-filter 에이전트 소환
    · True_Log (HIGH), Fail_Log (HIGH), MasterLog (MEDIUM) 읽기
    · → {FILTERED_DATA} 수신

  Step 3. 내부 데이터 충분성 평가
    · 2+ 관련 항목 → sufficient
    · 0-1 관련 항목 → thin
    · "No relevant data" → search-only

[Phase 2 — 토의 루프] (v0.9.8 — 16턴 포맷)

  오케스트레이터가 직접 실행. Observer 없음.

  토의 구조 (v0.9.8):
    세션 1 (탐색전): 3교환 6턴 + 최후의 진술 2턴 = 8턴 (FRESH 에이전트)
    세션 2 (공방전): 3교환 6턴 + 최후의 진술 2턴 = 8턴 (FRESH 에이전트)
    총 16턴 (항상 2세션 스트레이트 진행)
    세션 2 에이전트는 세션 1 최후의 진술을 초기 컨텍스트로 주입받음.
    불균형 보고: 세션 1 → 세션 2 전환 시에만 조건부 판단.

  교환 루프 (세션당):

    교환 1:
      1. Advocate 소환 (subagent_type: "cpas-sandbox:advocate")
         · MEMORY_CONTEXT + CURRENT_DIRECTION + 토의 주제 + "찬성 입장" 전달
         · 세션 2: + SESSION_2_BRIEFING (세션 1 최후의 진술 양측 텍스트)
      2. Advocate 원본 출력 저장 (태그 포함)
      3. 태그 라인 추출: [D-?] [R-?] [C-?] [A-?] [S-?]
      4. 태그/구분선 제거 → 순수 텍스트
      5. Skeptic 소환 (subagent_type: "cpas-sandbox:skeptic")
         · MEMORY_CONTEXT + 토의 주제 + Advocate 순수 텍스트
         + FILTERED_DATA + 충분성 평가 전달
         · 세션 2: + SESSION_2_BRIEFING (세션 1 최후의 진술 양측 텍스트)
      6. Skeptic 원본 출력 저장 (태그 포함)
      7. 태그 라인 추출
      8. 태그/구분선 제거 → 순수 텍스트 (다음 교환 Advocate용)

    교환 2-3:
      · Advocate resume (같은 agent_id) + Skeptic 순수 텍스트
      · Skeptic resume (같은 agent_id) + Advocate 순수 텍스트
      · 매 턴 태그 추출 + 원본 저장

  내부 데이터 무결성 검사 (교환 간):

    ▶ 대상: 내부 프로젝트 데이터 인용만 (MasterLog/True_Log/Fail_Log 엔트리)
      · 외부 WebSearch 인용은 대상 아님
      · 논리/품질 판단 아님 — "데이터 존재 여부" 확인만 (컴파일러 수준)

    ▶ 검사 흐름 (양방향 대칭 — 양쪽 에이전트 동일 적용):

      교환 N 완료 후:
        [1] 에이전트 A (턴 2N-1) 출력 스캔 → 내부 데이터 인용 감지
            → {FILTERED_DATA}와 대조 → 불일치 발견?
              → 에이전트 B (턴 2N)가 해당 오류 지적했나?
                → 지적함: 개입 안 함 (자정작용 정상 작동)
                → 못 잡음: 에이전트 A 다음 턴에 DATA_CHECK 대기열 추가

        [2] 에이전트 B (턴 2N) 출력 스캔 → 내부 데이터 인용 감지
            → {FILTERED_DATA}와 대조 → 불일치 발견?
              → PENDING 플래그 (교환 N+1에서 에이전트 A가 잡는지 확인)

      교환 N+1 시작:
        [3] 에이전트 A 응답 (턴 2N+1)
            → [1]의 DATA_CHECK가 있으면 resume 프롬프트에 삽입
            → 에이전트 A가 [2]의 PENDING 오류를 잡았나?
              → 잡음: 플래그 해제
              → 못 잡음: 에이전트 B 다음 턴에 DATA_CHECK 대기열 추가

        [4] 에이전트 B 응답 (턴 2N+2)
            → [3]의 DATA_CHECK가 있으면 resume 프롬프트에 삽입

    ▶ DATA_CHECK 삽입 형식 (오류 에이전트의 resume 프롬프트에 추가):

      ⚠ DATA CHECK: 이전 턴에서 {구체적 인용} 참조 —
      필터 출력 데이터에 해당 항목 없음.
      논리 구조는 유지하되, 존재하지 않는 내부 데이터 인용은 정정하여 계속하시오.
      없는 데이터에서 파생된 주장이 있다면 재점검하시오.

    ▶ 핵심 규칙:
      · DATA_CHECK는 오류 에이전트에게만 전달 — 상대에게는 안 보임
      · 상대가 이미 잡았으면 시스템 개입 안 함 (자정작용 우선)
      · 오케스트레이터는 {FILTERED_DATA}를 이미 보유 (Step 2) — 추가 비용 없음
      · 이것은 "내용 개입"이 아님 — O-2-4 위반 아님 (변수 선언 확인 수준)

    ▶ 설계 사유 (MasterLog [33] + 세션 #16 분석):
      · v0.9.4 Advocate 재설계(Accept→Redirect→Propose)로 토의 안정성 확보 (S-7)
      · 그러나 Accept 단계에서 Skeptic 인용을 무비판 수용 → Type-X 위험 증가
      · Skeptic 오류는 Advocate가 잡기 어려움 (역할 비대칭: 검증자 vs 추진자)
      · Advocate 오류는 Skeptic이 잘 잡음 (검증이 본업 — [22] 5/5, [31] T14)
      · 따라서 시스템 수준의 데이터 무결성 검사로 비대칭 보정

  최후의 진술 (세션당):

    해당 세션의 동일 에이전트를 resume한다 (fresh spawn 아님).

    Advocate 최후의 진술 (세션 1: 턴 7 / 세션 2: 턴 15):
      · resume: {해당 세션 advocate_agent_id}
      · 프롬프트에 최후의 진술 지시 주입:
        "[최후의 진술 단계] 이 세션의 마지막 턴입니다.
         자신의 핵심 논점 + 상대방의 유효한 논점을 종합하여 최종 입장을 진술하세요.
         반드시 WebSearch를 수행하여 최종 입장을 뒷받침하는 최신 근거를 확보하세요.
         태그 기준: C-10, R-1~4, S-7, A-7~11"

    Skeptic 최후의 진술 (세션 1: 턴 8 / 세션 2: 턴 16):
      · resume: {해당 세션 skeptic_agent_id}
      · 동일 최후의 진술 지시 주입

  세션 전환 (세션 1 → 세션 2):

    세션 1 완료 (8턴) 후:
      · 중간 분석: 세션 1의 8턴 태그 시계열 분석
      · 불균형 감지 (S/C 격차):
        - 한 쪽 S ≤ 4 단 1턴이라도 + 상대 S ≥ 10
        - 한 쪽 S-1 또는 S-2 도달 (항복/근접 항복)
        - 한 쪽 C ≤ 3 지속 + 상대 C ≥ 7 (근거 격차)
      · 불균형 시:
        - ⚠ 사용자 승인 필수: "Session 1 불균형 감지: {측} 열세. Session 2에서 지원할까요?"
        - 승인 시: 열세 측 세션 2 전체에 강화된 모델 활성화
        - 거부 시: 양측 표준 모델로 진행
      · 세션 1 최후의 진술 텍스트 추출 (태그 제거)
      · FRESH 에이전트로 세션 2 시작 (세션 1 최후의 진술을 SESSION_2_BRIEFING으로 주입)

  세션 2 완료 (턴 16) 후:
      · 전체 16턴 쟁점 구조화
      · 세션 1 궤적/결론 vs 세션 2 궤적/결론 비교
      · 세션 간 변화: 무엇이 바뀌었나? 무엇이 유지됐나?
